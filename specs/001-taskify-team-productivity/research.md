# 리서치: Taskify 팀 생산성 플랫폼

기술 컨텍스트에 있던 불확실한 항목과 주요 기술 선택에 대한 결정 사항을
정리한다. 명세(`spec.md`)에는 [NEEDS CLARIFICATION] 마커가 없었으므로,
여기서는 헌법과 기능 요구사항을 만족시키는 구체적인 기술 선택 근거만
다룬다.

## 1. 실시간 갱신 방식

**결정**: Socket.IO를 사용한 WebSocket 브로드캐스트(자동 폴백 포함).
프로젝트별 "room"에 클라이언트를 join시키고, 태스크 상태 변경
(`task:updated`)과 댓글 등록(`comment:created`) 이벤트를 같은 room의
모든 클라이언트에게 브로드캐스트한다.

**근거**: FR-007, FR-009, SC-002, SC-004가 요구하는 "새로고침 없이 5초
이내 반영"을 만족하려면 서버 푸시가 필요하다. Socket.IO는 WebSocket이
막힌 환경에서 롱폴링으로 자동 폴백하여 사내 네트워크 환경에서도
안정적으로 동작하며, Node.js/Express 백엔드와 통합이 간단하다. REST
엔드포인트는 계속 자원 변경의 단일 진실 공급원(source of truth)으로
남고, WebSocket은 "다른 클라이언트에게 다시 조회하라"는 알림 또는 최신
페이로드를 전달하는 보조 채널로만 사용해 헌법 원칙 IV(RESTful API
설계)를 해치지 않는다.

**검토한 대안**:
- Server-Sent Events(SSE): 서버→클라이언트 단방향만 지원해 구현은
  단순하지만, 댓글 작성 같은 양방향 상호작용에는 결국 REST와 병행해야
  하므로 이점이 크지 않음.
- 단순 폴링(주기적 GET): 구현이 가장 단순하지만 5초 이내 반영을
  보장하려면 폴링 주기를 짧게 잡아야 해 불필요한 서버 부하가 발생함.

## 2. 드래그 앤 드롭 라이브러리

**결정**: `@dnd-kit`

**근거**: 활발히 유지보수되고 있으며, 키보드 조작을 위한 센서를
기본 제공해 헌법 원칙 III(접근성 준수)이 요구하는 "완전한 키보드
탐색 가능성"을 라이브러리 차원에서 지원한다. React 18 및 TypeScript
strict 모드와 호환된다.

**검토한 대안**:
- `react-beautiful-dnd`: 널리 쓰였지만 유지보수가 중단되어 React 18
  strict mode에서 알려진 이슈가 있음.
- 네이티브 HTML5 Drag and Drop API: 브라우저 간 동작 차이가 크고
  키보드 접근성을 직접 구현해야 하는 부담이 큼.

## 3. 백엔드 프레임워크 및 저장소

**결정**: Express(TypeScript, strict) + Prisma ORM + SQLite 파일 DB.

**근거**: 헌법의 "기술 스택 및 도구" 섹션은 언어를 전체 코드베이스에서
TypeScript strict로 통일할 것을 요구하므로, 백엔드도 TypeScript로
작성한다. Express는 REST 라우팅에 특화된 가볍고 검증된 프레임워크다.
SQLite는 별도 DB 서버 없이 파일 하나로 동작해 "첫 테스트 버전"의
운영 부담을 최소화하며, Prisma 스키마는 이후 PostgreSQL 등으로
교체할 때도 마이그레이션 경로가 명확하다.

**검토한 대안**:
- 인메모리 저장(재시작 시 소실): FR-010(세션 간 영속성)을 만족하지
  못해 제외.
- PostgreSQL: 기능적으로는 충분하지만, 첫 테스트 버전 규모(사용자 5명)
  대비 별도 DB 서버 운영 비용이 과함.

## 4. 활성 사용자 식별(로그인 없음)

**결정**: 클라이언트가 앱 진입 시 5명의 사전 정의 사용자 목록에서
하나를 선택하고, 이후 모든 API 요청에 사용자 식별자(`X-User-Id` 헤더)를
함께 전달한다. 서버는 이 값을 신뢰하고(요청자 검증 없음) 요청의
작성자/행위자로 기록한다.

**근거**: 명세의 가정에 따라 이번 버전은 인증 메커니즘이 요구되지
않는, 고정된 5명 팀의 내부 테스트 도구다. 별도 자격 증명 없이 신원만
선택하는 방식이 FR-001과 SC-003을 가장 단순하게 만족시킨다.

**검토한 대안**:
- 세션 쿠키 기반 로그인: 명세에서 명시적으로 "로그인 기능 없음"을
  요구하므로 범위를 벗어남.

## 5. 테스트 및 커버리지 도구

**결정**: 프런트엔드는 Vitest + React Testing Library, 백엔드는
Vitest + Supertest. 두 워크스페이스 모두 v8 커버리지 provider를 사용해
CI에서 문장/분기/함수/라인 기준 80% 게이트(헌법 원칙 II)를 강제한다.

**근거**: Vitest는 TypeScript/ESM을 별도 트랜스파일 설정 없이 지원하며
frontend/backend 양쪽에서 동일한 도구로 통일할 수 있어 유지보수
비용을 줄인다.
